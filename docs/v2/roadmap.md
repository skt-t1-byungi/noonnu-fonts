## 로드맵(전략)

V2는 “대량 자동 수집”보다 “안전한 승인/배포 루프”를 먼저 완성하고, 그 다음 수집 소스를 확장하는 순서가 리스크가 낮다.

## Phase 1 — V2 최소 골격(2~5일)

- 레지스트리 스키마 확정(`data/v2/registry/fonts.json`)
- CLI(로컬/CI)로 다음이 가능:
  - 레지스트리 로드/검증
  - 후보 추가(수동 JSON ingest)
  - 증거 스냅샷 저장(HTTP fetch → file + sha256)
  - `reviewStatus` 기반으로 패키징 대상 필터링

산출:

- “동작하는 V2” (아직 자동 수집/LLM은 없어도 됨)

## Phase 2 — 품질 파이프라인 강화(1~2주)

- 폰트 파일 검증(파싱/이름/포맷)
- 변환(otf/ttf → woff2) 도입
- 패키지 포맷 고도화:
  - `meta.json`, `EVIDENCE.md`, README 템플릿 표준화
  - 예제 이미지 생성(렌더링 검증 포함)

## Phase 3 — 수집 소스 확장(지속)

우선순위(권장):

1) 공식 배포 페이지가 명확한 공공기관/지자체(라이선스 명시 가능성이 높음)
2) GitHub 릴리즈(버전/해시/릴리즈 노트 확보 쉬움)
3) 카탈로그/블로그(단, 라이선스 애매한 경우 많아 검토 비용 큼)

## Phase 4 — LLM 도입(선택, 안전장치 포함)

LLM은 “발견/요약/필드 추출”에 제한한다.

- 프롬프트는 버전 관리(문서/템플릿)하고,
- 출력은 반드시:
  - 근거 URL/스냅샷 경로를 포함
  - 스키마로 검증(Zod/JSON Schema)
  - 사람이 승인하기 전엔 배포 불가

## Phase 5 — 퍼블리싱 운영 고도화

- 주간 워크플로우:
  - 자동: Discover/Harvest/Auto-classify → PR 생성(또는 레지스트리 변경 제안)
  - 수동(승인): Package/Publish 실행
- deprecate 정책:
  - 라이선스 변경, 배포 중단, 취약/손상 파일 발견 시 처리 규칙
