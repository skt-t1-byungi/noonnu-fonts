# noonnu-fonts v2 리뉴얼: 문제정의(Problem Statement)

## 배경
이 저장소는 눈누(`https://noonnu.cc`)의 폰트 정보를 기반으로 폰트를 내려받고, 폰트별 npm 패키지(`@noonnu/*`)를 자동 생성/배포하는 프로젝트입니다.

## 현재 문제
- **단일 데이터 소스(눈누) 의존**: 눈누의 페이지/데이터 구조 변경에 따라 업데이트/배포 파이프라인이 자주 깨질 수 있음.
- **분류/네이밍의 비일관성**: 같은 패밀리(동일 폰트)임에도 두께(웨이트)별로 서로 다른 항목/패키지로 분리되거나, 반대로 다른 폰트가 하나로 합쳐지는 등 일관성이 부족함.
- **배포 단위의 불안정성**: “npm 패키지 1개”가 무엇을 의미하는지(패밀리/서브패밀리/웨이트)가 소스 데이터에 의해 흔들려 사용자 경험이 나빠짐.
- **업데이트 추적/동일성 판단의 어려움**: 폰트가 업데이트 되었는지, 동일 폰트인지(중복) 판단 기준이 약해 자동화 품질이 떨어짐.
- **리포지토리 용량/운영 문제**: 폰트 바이너리(woff/ttf/otf 등)를 git으로 직접 관리하면서 레포 용량이 불필요하게 커지고, 히스토리 관리가 어려움.

## v2 리뉴얼의 목표(Goal)
“눈누에 종속되지 않는” 자체 폰트 디깅/수집 파이프라인을 구축하고,
- “하나의 폰트(패밀리)”를 **사람의 커먼센스 결정 + 근거**로 정의하여 SoT로 누적하고,
- LLM으로 **모호한 케이스를 큐로 관리**하여,
**일관된 배포 단위와 안정적인 업데이트**를 제공한다.

## 비목표(Non-goals)
- 모든 폰트에 대해 라이선스를 100% 자동 판별/승인하는 것을 1차 목표로 두지 않는다. (사람 검토가 필요한 케이스를 큐로 분리)
- 처음부터 “완벽한 UI/검색 서비스”를 만들지 않는다. (CLI/매니페스트 중심으로 시작)
